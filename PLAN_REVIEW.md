# Adversarial Review — mcpp C++26 Design Plan (PLAN.md)

## 1. Executive Verdict

The plan is **architecturally sound and unusually disciplined for a design-stage document.** The core bets — BSP freeze/thaw as the single concurrency model, a static-slot sparse-mmap archive, f32-everywhere lossy codec, mcpp-owned geometry types — are coherent and mostly internally consistent. The adversarial pass downgraded the large majority of "CRITICAL/MAJOR" findings: many were arithmetically real but rested on misreadings of the plan's own mitigations (render-now-refine-later, the 2-bit occupancy oracle, persistent local mirror, per-block decode in the mutate phase). This is a plan that has already thought about most of its hard edges.

Two findings survived as genuinely CRITICAL or near-CRITICAL and **must be resolved before P1**, and a cluster of MAJOR integration-seam gaps threaten the project not because any one is fatal, but because they fall between subsystems that no part of the plan owns.

**The 3–5 things most threatening the project:**

1. **The hard τ (max-error) guarantee is unsound as written** (CRITICAL, codec). It is verified against the *encoder's own simulated decode* but the plan deliberately makes decode non-deterministic across ISAs. The user's decoder can breach τ silently while CI stays green. This breaks the single hard guarantee the codec advertises for surface/geometry data.
2. **GCC 15.2.0's libstdc++ ships no `<mdspan>`** (CRITICAL, feasibility). `std::mdspan` is the spine type (volume_view, occupancy, Grid, DenseView), and both-compiler build is a hard merge gate. The codebase cannot pass its own CI on the very first foundational type. Cheap to fix, but blocking today.
3. **The VC3D migration is an unbounded, un-sized, all-or-nothing flag day** (MAJOR, testing/integration). The "no shims" stance means VC3D cannot run half-on-mcpp, there is no end-to-end acceptance test, and the codec non-determinism removes the old golden-output oracle. This is the largest unestimated risk in the project.
4. **Integration-seam gaps no subsystem owns** (MAJOR ×several): on-disk byte-order/endianness is unspecified for an explicitly cross-machine format; BSP "composition" is actually a nested barrier tree the plan describes as flat; there is no whole-process memory budget; failure propagation (`abort()` vs `std::expected` vs infinite-loop) is incoherent for a library embedded in a long-running GUI.
5. **Streaming has no progress guarantee under flaky/changing remotes** (MAJOR ×2): transient S3 failure → positive-feedback re-request storm with no backoff and no terminal state (offline loop can spin forever); and the up-front occupancy map is trusted forever with no re-validation, yielding silent stale-air holes (missing papyrus) on a re-segmented volume.

## 2. Findings by Adjusted Severity

### CRITICAL

#### C1. Hard τ bound holds only on the encoder's simulated decode, not the user's decoder (codec) — **REAL**
- **Claim:** "τ>0 ⇒ |orig−decoded| ≤ τ on every material voxel, guaranteed and self-contained (decoder needs no τ)... survives fast-math" (PLAN.md:72, 227, 1051, 1059).
- **Failure scenario:** Residual correction is a DCT-coded f32 residual reconstructed through the same inverse-quant + inverse-f32-DCT the plan declares non-deterministic across ISA/opt/fast-math (PLAN.md:78, 996, 1064). Encoder on AVX2 achieves |orig − decoded_AVX2| ≤ τ; user decodes on NEON (or different `-ffp-contract`) → decoded' = decoded_AVX2 ± δ → |orig − decoded'| ≤ τ + δ. The corrections were computed against a decode the user never runs. The guarantee is self-*referential*, not self-*contained*.
- **Verifier verdict: REAL, CRITICAL upheld.** The "integer-mask-anchored / survives fast-math" justification is a category error — it conflates the residual correction with the *air mask* (the air mask genuinely survives fast-math because air is forced to exactly 0 by an integer compare outside the float path; no equivalent post-decode integer override exists for the residual). The original codec's guarantee held only because it used an *integer* DCT; mcpp deleted the integer path but carried the guarantee's wording forward without re-deriving it. τ≈0.05–0.1 voxel is a subvoxel tolerance whose fractional part IS the trilinear interpolation weight; a breach corrupts downstream segmentation/flattening. Caveat: whether δ actually exceeds the τ margin on a given residual is unproven without a CONFIRM experiment — but no guard band exists, so the guarantee is unsound as stated.
- **Fix:** Pick one: (a) make decode bit-deterministic for case-4 surfaces only (`-ffp-contract=off`, fixed evaluation order on that kernel — accept the perf hit on the non-hot surface path); or (b) widen the encode-side bound to τ−δ_max where δ_max is the measured worst-case cross-ISA decode divergence; or (c) store the final correction as an exact post-decode additive integer delta applied outside the float transform (mask-style). Stop claiming "guaranteed" + "survives fast-math" simultaneously — they are contradictory as written.

#### C2. GCC 15.2.0 ships no `<mdspan>`, but mdspan is the foundational type everywhere (feasibility) — **REAL**
- **Claim:** volume_view/Grid/occupancy/DenseView are all `std::mdspan` (PLAN.md:114-118, 308, 398, 747, 844, 949); both-compiler build {Clang 21, GCC 15} is a hard merge gate (PLAN.md:1081).
- **Failure scenario:** `#include <mdspan>` reproduced on this box: compiles on clang++-21 -stdlib=libc++; fails on g++-15 with "fatal error: mdspan: No such file or directory". `/usr/include/c++/15/mdspan` does not exist; `__cpp_lib_mdspan` undefined. `mcpp::zfast_layout` is only a layout *policy* (third template arg) — it still requires `<mdspan>`, so it is no escape. The first foundational TU fails the gate immediately.
- **Verifier verdict: REAL, CRITICAL upheld.** Reproduced on the exact mandated toolchain (g++ 15.2.0). No vendored/Kokkos fallback exists anywhere in plan or memory; reliance on `std::mdspan` is unconditional and undocumented as a risk. Mitigating note: it is a dependency/build-config issue with cheap fixes, not a design defect — but as written the plan is blocked.
- **Fix:** Vendor the reference Kokkos mdspan (header-only) as `mcpp::mdspan` unconditionally (this also sidesteps the libc++/libstdc++ divergence), OR drop/pin GCC until libstdc++ ships mdspan. Do not assume "GCC 15 = C++23 library complete."

### MAJOR

#### M1. A needed fine block can be evicted the same tick it is filled; offline loop-to-completion can livelock (cache) — **REAL (downgraded CRITICAL→MAJOR)**
- **Claim:** apply() order is insert-fresh → bump-requested → generation-sweep evict (PLAN.md:531-533, 540).
- **Failure scenario:** Generation stamp = tick counter, so *all* blocks touched/filled in tick T share generation T and are indistinguishable; "oldest-first" is undefined among equal generations. When working set W > capacity N, the sweep must evict generation-T blocks — including ones the next read phase needs. Offline mode "loops until the miss-list is empty" (PLAN.md:726) → never empties → infinite loop.
- **Verifier verdict: REAL.** No pin/no-evict set and no W≤N bound exist anywhere (grep-confirmed). §3.17 explicitly defers eviction-cutoff and sizing math. Downgraded to MAJOR because: it's textbook fixed-capacity-cache behavior that only livelocks due to the one-line "until empty" promise; the interactive path is partially protected by get_best_lod coarse fallback; and eviction policy is openly an open item. NRU-vs-LRU is a red herring.
- **Fix:** (1) Within apply(), never evict any block in the current tick's interest set (pass it in as a pin set); evict only strictly-older generations. (2) Add an invariant + test: per-frame distinct-block working set ≤ Capacity, with a defined failure mode (global drop to coarser LOD) when violated. Tie N to the largest supported frame working set.

#### M2. Fixed slot-stride forces TB-to-EB logical files; the "~200GB logical" figure is wrong (archive) — **REAL (downgraded CRITICAL→MAJOR)**
- **Claim:** "logical file size ~200GB" with whole-file sparse mmap (PLAN.md:290; mcpp-archive.md:27).
- **Failure scenario:** Static slots require worst-case stride. Stride: u8=16.8MB, u16=33.6MB, f32=67.1MB. Realistic 14000³ u8 scroll = 2.8TB; f32 = 11.2TB; single LOD0 at the 2^20 cap = 1.15 EB. The "~200GB" figure corresponds only to a ~5800³ u8 cube.
- **Verifier verdict: REAL arithmetic, MAJOR (not CRITICAL).** A 2.8–11TB sparse mmap is fully viable on the Linux x86-64 target (128TB user VA); it is the *same* engineering problem at larger scale. Within-slot padding lands in sparse holes (zero real disk). The genuine ceiling is the 2^20 cap (1.15 EB exceeds VA, and is un-creatable on any real filesystem anyway). So: a wrong sizing figure plus an undocumented cap-case ceiling, not a design showstopper.
- **Fix:** Correct the figure to "hundreds-of-GB to low-TB common case, EB at the cap." Add an explicit per-LOD windowed-mmap note for large files and state the max mmappable logical size.

#### M3. Oversize-chunk `abort()` WILL fire on incompressible content / tight-τ residuals (archive) — **REAL**
- **Claim:** "codec is lossy-and-shrinking ⇒ compressed ≤ raw always... `if (payload_len > raw_chunk_bytes) abort();` (release too)" (PLAN.md:368-370).
- **Failure scenario:** DCT+entropy does not guarantee output ≤ input. An adaptive range coder on max-entropy content emits ~input + model overhead + the 16404-byte header; for a u8 chunk (raw = 16 MiB, ~0.1% header slack) this can cross raw size. Tight-τ residual streams on noisy micro-CT are the most concrete trigger. There is no store-raw escape anywhere in the format (grep-confirmed; blocks are strictly 2-state ALL_ZERO vs REAL). One adversarial chunk aborts a multi-day 100TB ingest with no fallback.
- **Verifier verdict: REAL, MAJOR.** Scenario-3 (f32/u32 light quant) is effectively refuted — f32 raw = 64 MiB gives huge header headroom. The tight case is narrowly u8. Real scroll CT is air-dominated, so the trigger is adversarial/edge (reconstruction-edge noise, partial scans), not the common path; overshoot is small and bounded. Stays MAJOR purely for blast radius (days-long ingest, zero fallback).
- **Fix:** Replace `abort()` with a per-block STORE-RAW fallback (flag bit; compressed≥raw → store raw). Size slot stride to raw + worst-case header so store-raw always fits. Keep the assert only as a post-store-raw corruption check. Standard in every production codec.

#### M4. Streaming has no progress guarantee under flaky S3 (streaming) — **REAL**
- **Claim:** "Transient failure leaves the chunk DONT_KNOW... re-requested next tick... zero retry machinery, no backoff" (PLAN.md:677, 698).
- **Failure scenario:** Under S3 503 SlowDown, the coordinator re-requests the same failing set at the same fan-out every tick — positive-feedback overload precisely when S3 asks for less. Offline loop-to-completion never terminates; a permanently-unfetchable chunk (404/403/corrupt) marked REAL in the up-front map stays DONT_KNOW forever with no surfaced error. M6's "corrupt-chunk scrub" feeds the infinite loop rather than ending it.
- **Verifier verdict: REAL, MAJOR.** No backoff/terminal-FAILED state exists; backoff is delegated to a coordinator that is never specified to do it. Mitigated: the ingest path is immune (no DONT_KNOW chunks); interactive degrades to holes rather than hanging; the permanent-404 case requires remote inconsistency. Fix is additive within BSP.
- **Fix:** Add a per-chunk failure counter + terminal FAILED disposition in the coordinator's request set (not the persisted 2-bit map): N transient failures → exponential backoff with jitter; 4xx → immediate terminal failure surfaced via `std::expected` so the offline loop exits instead of spinning. A few integers per in-flight chunk, BSP-compatible.

#### M5. Up-front occupancy map is trusted forever → silent stale-air holes (streaming) — **REAL**
- **Claim:** "The map is assumed stable for the session... no mid-session re-fetch is planned"; "air chunks cost ZERO network, ever" (PLAN.md:630; mcpp-streaming.md:22).
- **Failure scenario:** Vesuvius volumes are re-segmented / ROI-masks widened / re-uploaded. A long-running session opens with a snapshot, the remote widens its mask (former air → REAL), and the client never re-fetches those chunks — silently rendering/sampling them as empty (missing papyrus), with no detection path. Persists across runs (the local file *is* the cache).
- **Verifier verdict: REAL, MAJOR.** No versioning/ETag/content-hash/TTL pin exists (grep-confirmed); mcpp-streaming.md:28 lists re-fetch as an *unresolved* open question, not a mitigation — so the assumption is asserted, not guaranteed. The secondary "dims grew → wrong indexing" sub-claim is weaker (header+map fetched together at open). MAJOR (not CRITICAL) because it requires the session to span a remote update.
- **Fix:** Capture the remote object's ETag/version-id + header format_version/dims at bootstrap; store it in the local .mcp header; re-validate cheaply (HEAD/If-None-Match) on an interval or on any chunk xxh3 mismatch, invalidating affected occupancy back to DONT_KNOW. At minimum assert-die on version mismatch at open.

#### M6. Write-through local mirror has no multi-process or read-only-mount story (streaming) — **REAL**
- **Claim:** BSP's no-locks guarantee + "the local file IS the cache of remote, persistent across runs" + "NO flush-ordering guarantees."
- **Failure scenario:** Running viewer + tracer + ink-detection against one shared mirror (normal Vesuvius pattern to avoid re-downloading 100TB+) is unprotected: dedup is intra-process only, so two processes pwrite the same slot with no lock and no flush ordering, and an occupancy flip written separately from payload lets a third reader observe REAL over a half-written slot. A read-only NFS/SMB golden-dataset mount cannot serve as the mirror at all (open() needs write access).
- **Verifier verdict: REAL, MAJOR.** No flock/exclusive/single-process precondition and no read-only mode exist (grep-confirmed). Tempered: both processes fetch identical bytes from the same remote (torn window is narrow, not divergent-data corruption); NFS/SMB sparse semantics are partially acknowledged ("detect+warn/fallback", though fallback is undefined). MAJOR because the multi-client and RO-mount cases are normal deployments and a single advisory flock is near-free.
- **Fix:** Take an exclusive advisory `flock(LOCK_EX)` on the mirror at open (fail-fast if held), OR design a multi-process-safe publish (occupancy flip strictly after `msync` of payload, with a per-chunk "writing" intent bit). Add an explicit read-only-mirror mode that serves only present chunks and errors on misses. Detect NFS/SMB and refuse sparse-mmap rather than silently corrupting.

#### M7. pointTo is a non-convex local search with no convergence-quality regression gate (geometry) — **REAL**
- **Claim:** pointTo = "nearest-surface-point gradient-descent search" (PLAN.md:882), hint-driven by the patch index, under f32+fast-math, with "zero golden-byte tests" posture.
- **Failure scenario:** Under fast-math + a from-scratch tree whose "nearest" returns a different (still-valid) candidate, pointTo can converge to a different local minimum or fail its threshold (tracer wanders, produces holes, loops to max_iters). This is invisible to PSNR/tolerance gates — there is no geometry-convergence gate proposed.
- **Verifier verdict: REAL, MAJOR.** Confirmed: P6 (PLAN.md:1105) golden-gates only the *deterministic* coordinate math; nothing bounds pointTo's converged-distance distribution. The detailed fragility mechanics reference VC3D source not in this repo, so are asserted not verified here; tracer-side VC3D tests are a partial secondary net (outside mcpp CI). Fix is conventional and cheap.
- **Fix:** Add a geometry-convergence gate — on a real-surface corpus, assert mcpp pointTo converged-distance and success-rate are within a bound of the reference. Pin pointTo's step/iter schedule as a spec, and decide whether the minimizer may use fast-math at all.

#### M8. Tolerance-only codec gates can't catch in-band quality regression (testing) — **REAL (downgraded CRITICAL→MAJOR)**
- **Claim:** "ZERO golden-byte codec tests... within PSNR/MAE/max-error/SSIM bounds"; product bar is "iso-rate PSNR leads c3d +1.4–3.7 dB."
- **Failure scenario:** `EXPECT_PSNR_GE(decoded, original, 42.0)` is a one-sided absolute floor. A refactor dropping quality 47→43 dB at iso-rate (or inflating bitrate 15%) passes every gate while the codec regresses below its entire reason to exist. The product value is a *relative*, two-axis (rate AND distortion) claim that no gate tests.
- **Verifier verdict: REAL, MAJOR.** Confirmed: no RD-regression gate, no checked-in per-(dtype,usecase,quality) RD baseline, no bpv ceiling, no c3d-relative delta anywhere; the differential test vs the original C codec is also tolerance-based. Downgraded because §8.8 already flags gating benchmarks/threshold as open ("a regression here would silently weaken the suite") — partial awareness — and no suite exists yet. PSNR/bpv metric *values* are stable to ~0.01 dB under reassociation, so a checked-in RD baseline is straightforward despite non-deterministic bytes.
- **Fix:** Replace the absolute floor with a relative RD regression gate: check in per-(dtype×usecase×quality) {bits-per-voxel, PSNR/MAE/SSIM} baselines on a fixed corpus, gate on tight two-sided deltas (e.g. PSNR within −0.1 dB AND rate within +1%), plus an explicit c3d-lead-margin gate.

#### M9. Cross-ISA tolerance threshold is underived and validated against QEMU (testing) — **REAL**
- **Claim:** "AVX2 vs NEON via QEMU: both must decode within tolerance... never byte-identical" (PLAN.md:1064), a hard merge gate.
- **Failure scenario:** No tolerance value is derived anywhere. Set loose → a NEON-only bug below the slack passes undetected (NEON is first-class). Set tight → normal fast-math divergence flakes. Worse, QEMU user-mode softfloat does not reproduce hardware NEON FMA/rounding/FTZ under fast-math — the gate validates ARM against a non-faithful emulator.
- **Verifier verdict: REAL, MAJOR.** Confirmed: the cross-ISA tolerance value appears nowhere (not even in the "still open" lists — an *unrecognized* gap), the loose/tight dilemma is self-inflicted by f32+fast-math+compile-time lane width, and the QEMU-fidelity problem is unaddressed. The per-ISA roundtrip test is a second net but is itself an undefined tolerance.
- **Fix:** Acquire real aarch64 CI hardware. Derive tolerance from the DCT's conditioning, not a guessed number. Better: gate cross-ISA on the *quantized integer coefficient indices* (which should match near-exactly) rather than reconstructed f32 — a near-exact integer compare that localizes ISA bugs.

#### M10. VC3D migration is an unbounded, un-sized, all-or-nothing flag day (testing/integration) — **REAL**
- **Claim:** "totally greenfield... NO IChunkedArray shim, VC3D will be modified to call mcpp"; consumers convert at their boundary (PLAN.md:25, 61, 841).
- **Failure scenario:** The migration is simultaneously (a) rewrite every cv::-typed call site in tracer/segmentation/flattening/ABF/GrowSurface/GrowPatch, (b) convert VC3D's synchronous "fetch returns full data" to BSP's render-now-refine-later, and (c) re-validate outputs against a now-f32-non-deterministic decode that voids the old golden comparison. No shim → VC3D cannot run half-on-mcpp; no end-to-end acceptance test; no migration milestones. The plan never sizes this or defines done.
- **Verifier verdict: REAL, MAJOR.** Confirmed: no-shims is explicit and repeated; all milestone lists terminate at mcpp's boundary; the only cross-system test is the codec differential. Tempered: case-4 max-error bounds + exact coordinate-math goldens mean a tolerance-based geometry-equivalence test is *constructible* (oracle reduced, not eliminated); VC3D is a separate repo arguably at the edge of this plan's scope — but the plan uses the integration to *justify* building the whole geometry layer, so leaving it un-sized is load-bearing. MAJOR not CRITICAL.
- **Fix:** Define a VC3D-on-mcpp end-to-end acceptance test (real tracer/segment/flatten pass, compare surfaces within geometric tolerance) as the definition of done. Reconsider absolute "no shims" — a temporary IChunkedArray-shaped adapter over the BSP coordinator enables incremental, validated cutover. Add migration milestones with runnable intermediate states.

### Notable MINOR / QUESTION

- **[MINOR] BSP "one superstep shape" prose is looser than the pipelined reality (bsp-architecture).** IO genuinely spans multiple ticks (PLAN.md:548/550 "gathered during the PREVIOUS tick"; dedup against prior-tick in-flight), which is in real tension with "no writers during read." But the headline frame-stall and data-race claims are **REFUTED**: the frame never blocks on fetch (render-now-refine-later is the adopted primary model, PLAN.md:917, 514-516), and IO workers write disjoint not-yet-occupied slots flipped only at the barrier. *Fix:* clarify in prose that the mutate/IO phase issues fetches harvested at a later thaw (pipeline depth), and that "no writers during read" means no writer to any slot a frozen reader can observe.
- **[MINOR] The coordinator is an under-specified god-object (bsp + cache).** Real, concurring across two dimensions: the coordinator is referenced ~15× but has NO module, section, API, or milestone of its own, and the cross-tick in-flight set has no named owner/lifetime/retry-reconciliation (DONT_KNOW conflates "never requested" with "in flight"). But the "serial Amdahl wall caps speedup under 16 cores" claim is **REFUTED** — it mis-assigns the dominant decode cost to the serial path; decode/classify run on parallel IO workers, leaving only cheap O(misses)×~100ns gather/dedup/apply serial. *Fix:* give the coordinator a first-class subsystem/API/milestone; make the in-flight set a bounded, tested structure (or add an IN_FLIGHT occupancy state); bound and benchmark per-tick serial coordinator time.
- **[MINOR] Static band partitioning has structural tail latency (bsp).** Real (join barrier waits for the slowest band; no work-stealing). But the "whole-chunk decode in bands → 10–50×" magnitude is **REFUTED** — decode is per-missed-block in the coordinator's mutate phase, not in render bands; finer static tiles already mitigate without stealing. *Fix:* note tile-granularity as a tuning concern; optionally bless one atomic tile counter for dynamic assignment.
- **[MINOR] "No atomics-as-sync, NONE" is contradicted by the barrier itself (bsp).** The physics is right (a publishing barrier needs a real acq/rel edge), but the self-contradiction is **OVERSTATED** — the plan isolates threading primitives in `mcpp::os` (PLAN.md:158) and scopes "NONE" to concurrent-mutation apparatus. *Fix:* reword the invariant honestly: "all cross-thread sync is confined to ONE audited primitive — the phase barrier — in mcpp::os; no application/subsystem code contains synchronization." Mandate a standard-library barrier.
- **[MINOR] Read-phase immutability vs lazy surface caches (bsp/geometry).** Page-fault leg **REFUTED** (PLAN.md:366 explicitly accepts demand paging as OS-internal, not mcpp sync). Lazy QuadSurface caches (validMask/normals/ensureLoaded) shared across band workers under `const surf&` are a real but minor unstated-discipline gap. *Fix:* mandate materializing all lazy surface state in the mutate phase before handing a shared const view to band workers (the same discipline already stated for spatial trees).
- **[MINOR] Whole-chunk-always fetch over-fetches for oblique slices (streaming).** Real tradeoff (~16× per material chunk), but the "50–100× / defeats the perf floor" framing is **OVERSTATED** — the 96× figure was a network metric explicitly renounced, not coupled to the compute floor; air chunks cost zero network (up-front occupancy map); persistent mirror amortizes to ~1× per session; get_best_lod hides cold latency. *Fix (optional):* restore optional per-block ranged GET for the interactive consumer using the offset table; keep whole-chunk for the offline loop. Worth measuring first-touch latency on a real material-dense oblique slice.
- **[MINOR] u32/s32 lose low bits through f32; max-error mode can't bound them (codec).** Real and unacknowledged (f32 24-bit significand; values >2^24 truncated at ingest, no "24-bit" caveat anywhere). But **OVERSTATED** to MAJOR — the codec is admittedly lossy, and the affected value range has no real use in u8/u16 micro-CT. *Fix:* document the 24-bit precision bound (and that τ in integer units can't go below the f32 ULP at the magnitude), or restrict u32/s32 as u64/f64 were deferred, or declare max-error mode unavailable for u32/s32.
- **[MINOR] consteval tables (IEEE-strict) vs fast-math runtime (codec).** Numeric fact is true; all alleged consequences **REFUTED** — quant bins are transmitted as integers (no decode re-derivation), priors are symmetric shared consteval tables on both sides of one build with an adaptive coder, and τ is a measured encode-side loop. *Fix (one line):* never recompute at decode a quant/EOB threshold the encoder derived at compile time (design already does this); pin `-ffp-contract` for any expression compared against a runtime float.
- **[MINOR] Air-collapse anti-direction under fast-math (codec).** The catastrophic "material→0→reclassified as air→vanishes from LODs" is **REFUTED** — air-ness is a separate integer mask, decimation propagates the zero-*mask* (nonzero *children* = occupied per mask), and `max(x,1.0f)` admits no reassociation to 0. Real residual: the clamp ordering is underspecified. *Fix:* pin the decode tail as a strict integer post-pass — round f32→int, integer mask forces air→0, integer max(v,low) forces material away from 0, all outside any float op.
- **[MINOR] Compile-time capacity vs "adapt to VC3D memory pressure" (cache).** The headline contradiction is **REFUTED** — the cited "adapt to memory pressure" mandate is fabricated (grep-confirmed; mcpp is greenfield, VC3D is "reference, not a contract"); `Cache<T,Capacity>` is a template, monomorphizable per detected RAM. Residual MINOR: no in-place grow and no RAM give-back (no MADV_FREE). *Fix:* make capacity a runtime construction parameter (immutable during freeze); optionally a thaw-phase shrink hook driven by host memory pressure.
- **[MINOR] Open-addressed map full-behavior / zero-tag fill (cache).** The "zero-tag entries never reclaimed → probe chains → O(N)" core is **REFUTED** — generation lives in the *map entry*, so zero-tag entries age out by the same sweep (the finding's own refute condition). Real residual: the stated sizing "map = N/load_factor" ties map capacity to slab count, but zero-tags decouple them; size for peak live keys instead. Already flagged as an open sizing detail (PLAN.md:591). *Fix:* size the map for max live keys (slab-backed + zero-tag); add a 0.9+ load-factor perf test on a mostly-air volume.
- **[MINOR] Two-tick refine latency / convergence (cache).** Real kernel: the plan documents *contradictory* latencies — "previous tick" (PLAN.md:550, 2-tick) vs within-mutate-phase fetch+decode+apply (PLAN.md:670, 1-tick). The unbounded-convergence/lost-fill half is **REFUTED** (dedup is fetch-only; occupancy flips only after fetch lands; cache decode-misses are re-recorded every tick). *Fix:* collapse local-decode misses to one tick (gather→decode→apply in the same mutate phase); reserve multi-tick latency for genuine network fetches; specify that a block stays re-requestable until actually present.
- **[MINOR] Cold-pan first frame is holey despite "never holey" (geometry).** Real but narrow — "never holey" is literally false on a cold/never-visited region until the coarse pyramid is resident. **OVERSTATED** to CRITICAL: the tiny whole-scroll coarse pyramid loads up front (PLAN.md:276), so steady-state pans usually have a resident coarse block; synchronous in-frame fetch (the implied alternative) would be strictly worse. *Fix:* reword to "never holey once the coarse pyramid is resident"; note frustum-prefetch as a future hook; quantify cold-pan latency vs the 16–33ms budget.
- **[MINOR] cv↔mcpp conversion boundary (geometry).** The "Ceres residual blocks index Grid<Vec3f>, ~1100 sites, no thin boundary" mechanism is **REFUTED** by the actual code — the optimizer's parameter blocks point at tracer-internal scratch (a Vec2d map + a local Vec3d Mat), not the QuadSurface grid; QuadSurface enters loss functors read-only via rawPoints(); the boundary IS at construction, as planned. Real residual: porting GrowSurface/GrowPatch/ABF to read mcpp grids is genuine work, deliberately externalized to the VC3D side (ties to M10).
- **[MINOR] QuadSurface op-set completeness (geometry).** Mostly **OVERSTATED** — multi-component gen and the cv-op reimplementations are already in-scope/acknowledged. One genuine clean gap: **tricubic sampling** exists in the production composite path (Slicing.cpp:412 `SampleMode{Nearest,Trilinear,Tricubic}`) but is absent from mcpp's `Filter{nearest,trilinear}`. *Fix:* add tricubic to the Filter enum (or justify dropping with quality data); add a from-grid/sub-view constructor to the Surface concept.
- **[MINOR] "TSan finds nothing" is not a proof (testing).** Headline (tautology / structurally-incapable / atomics-banned-so-no-publish) largely **REFUTED** — the barrier (not banned) carries the happens-before edge; TSan models standard barriers and *would* catch a straggler-reader-vs-apply race. Real residual: a partition-boundary off-by-one is only caught if the overlapping write actually executes at runtime, and no adversarial partition/barrier-timing fuzzer is planned. "Races are IMPOSSIBLE" (PLAN.md:55) is overstated. *Fix:* add adversarial partition-decomposition fuzzing + straggler-reader fault injection under TSan and a deterministic interleaving harness; treat TSan as backstop, not proof.
- **[MINOR] AFL++ on a tolerance codec (testing).** "Invariants assigned to the wrong tool" is **REFUTED** — the τ and air-bit-exactness invariants are assigned to the exact/property-test layer (PLAN.md:8.2, P1), not AFL++; the roundtrip harness aborts on tolerance violation (so the property *is* the oracle). Real residual: no adversarial generators (τ-boundary, air/material edges) are specified — already an acknowledged open item (§8.8). *Fix:* add domain-aware generators to the property layer; keep AFL++ for crash-safety on untrusted bytes.
- **[MINOR] Perf-regression change-point gate vs measurement noise (testing).** Real tension (cold single-block decode + tight loops are the noisiest things to gate; PGO+ThinLTO shifts layout per profile), but **OVERSTATED** — the plan only commits to "change-point on a frequency-pinned runner, method TBD," fully compatible with the finding's own refute path (perf counters / distribution+CI). Defensible criticism: §8.8 mislabels this methodology-critical item as "minor, none blocking" when the hard gate's enforceability depends on it. *Fix:* gate per-PR on deterministic proxies (instruction/cache-miss counts); reserve wall-clock change-point for nightly trends on dedicated hardware; decide method/threshold before calling it a hard gate.
- **[MINOR] Greenfield rewrite with no RD oracle (testing).** Real verification hole (no relative RD-curve gate vs original/c3d), but **OVERSTATED** — partially mitigated by absolute PSNR/SSIM floors + a ±5% rate gate (PLAN.md:247). Overlaps M8. *Fix:* the original's RD *curve* is stable/checkable even with non-deterministic bytes — gate that mcpp meets-or-beats the original's PSNR at matched bitrate at every quality rung; keep the original codec buildable in CI as the live RD oracle.

### Findings explicitly REFUTED / DISMISSED (do not act on as bugs)
- **[DISMISS] DONT_KNOW→REAL stale zero-alias "black hole" (cache).** Both preconditions are foreclosed: no ZeroTag is ever emitted for a DONT_KNOW chunk (classification routes DONT_KNOW through fetch, PLAN.md:525), and a real FreshPair unconditionally updates the key's slot (PLAN.md:540). DONT_KNOW is purely transient. *Optional MINOR:* add a regression test (air-at-T, material-after-fetch must read as material) for defense-in-depth.
- **[DISMISS] SurfacePatchIndex can't be a static build-once tree (geometry).** Verified against real VC3D source: the batch tracer (GrowSurface/GrowPatch) rebuilds the index exactly once then queries read-only — already the PLAN §6 model; the incremental updateSurfaceRegion API is used only by the interactive GUI, which is out of mcpp scope. The alleged O(N²) reindex does not occur. *Residual (not a bug):* mcpp must still port the triangle/barycentric query surface — a feature-completeness item.
- **[REFUTED] Per-block page-alignment contradicts dense-pack (archive).** Misreading — the plan page-aligns only the chunk base and payload *start*, then dense-packs blocks; the "neighbor's tail" parenthetical means neighbor *chunk*. No 13.6× inflation; decode uses the offset table, not page boundaries. *Fix (one word):* "neighbor" → "neighbor chunk."
- **[REFUTED] SEEK_HOLE as occupancy oracle breaks on ZFS/NFS (archive).** The authoritative oracle is the application 2-bit occupancy map (PLAN.md:398), not SEEK_HOLE (used only as an optional reclaim/diagnostic hint). ZFS/NFS hole-detection quirks don't affect occupancy or decode. *Residual:* the "fallback" wording is undefined — restrict supported FS or design the actual fallback, and never rely on SEEK_HOLE for correctness.
- **[MINOR, not the claimed defect] Crash-safety "torn writes acceptable" (archive).** Technical mechanics correct (no write ordering; a multi-page chunk *can* tear; read path doesn't validate xxh3), but "indefensible" **OVERSTATED** for a deliberately-owned lowest-priority decision with bounded blast radius and structurally-idempotent re-run. *Fix:* one ordering rule (payload+table, then fsync, then flip occupancy as the single commit point) + a cheap per-chunk header magic/length sanity check on the read path; document that a crashed ingest may contain torn REAL chunks undetected on the hot path.

## 3. Cross-Cutting / Missed Issues (no subsystem owns these)

These span integration seams and are the most likely to bite *because* every dimension-critic stayed inside their subsystem.

- **[MAJOR] On-disk byte-order/endianness is never specified for an explicitly cross-machine format.** The plan defines exact integer on-disk structures (file header, the 16404-byte chunk header `[xxh3:u64][quality:f32][flags:u32][offset_table:4097×u32]`, the 2-bit occupancy map, the packed wire `PackedChunkHeader{u32 magic; u32 block_count; u32 offset_table[]}`) and reads them zero-copy via `bit_cast` off mmap (PLAN.md:405). Endianness appears nowhere. The codec's "drop cross-ISA byte identity" reasoning covers the *codec stream*, NOT the *container's integer offset tables / magic / dims / structural xxh3*. A packed `.mcp` produced on one host and fetched by a differently-endian client reads garbage offsets — invisible in single-host CI, surfaces only in production cross-host fetch. **Fix:** specify a canonical on-disk byte order (little-endian) for all integer fields and byte-swap on read for foreign hosts, or assert host endianness at open.

- **[MAJOR] BSP "composition" is a barrier *tree*, described as flat.** A render tick's MUTATE phase *contains* the streaming superstep (fetch GETs), the codec superstep (parallel per-block decode), the cache superstep (apply), and a spatial-tree rebuild — i.e. ≥3 nested fork-join sub-phases inside one tick's single mutate phase. The architecture memo claims "ONE mental model... a single barrier as the ONLY synchronization point." The plan never says how barriers nest, whether nested phases share one thread pool, who owns the thread count, or whether sub-phase barriers count as "the only sync point." The real model is recursive, not flat. **Fix:** specify the barrier-nesting/composition rules and thread-pool sharing explicitly; either embrace "BSP at multiple clock rates with a defined cross-loop handoff" or prove the sub-phases collapse into one.

- **[MAJOR] No whole-process memory budget; subsystems sized in isolation.** Cache slabs (heap, fixed N), archive sparse mmap (TB–EB), ~18 GiB occupancy map, streaming mirror mmap, per-tick spatial trees, per-band scratch/miss buffers, and lazily-loaded resident surfaces are each sized alone; nothing sums them or defines a total-process RSS target, a global pressure signal, or arbitration when they compete. The mmap page cache (faulted archive + mirror) and the heap slab cache contend for the same physical RAM with no coordination — under pressure the kernel may evict faulted archive pages the renderer is about to re-read while cold heap slabs are retained (cache has no MADV_FREE). VC3D also runs multiple volumes + a tracer alongside. **Fix:** define a steady-state working-set budget and a global memory-pressure arbitration policy across mcpp's own subsystems.

- **[MAJOR] Failure/abort propagation is incoherent for a library embedded in a long-running GUI.** Three incompatible models collide: `abort()` "in release too" (PLAN.md:266, 370), `std::expected<T,error>` at boundaries (PLAN.md:130), and streaming's no-terminal-state infinite loop (M4). A library that calls `abort()` on one corrupt chunk or oversize edge case takes down the host VC3D process and the user's unsaved tracing work; the fuzz-hardening (§8.5) covers untrusted *bytes* but not these *internal-invariant* aborts. **Fix:** define mcpp's library failure contract — internal-invariant violations should return `std::expected` (or a dedicated fatal-but-catchable error) rather than `abort()`, so the host can degrade instead of dying. (Note: M3's store-raw fix removes the most likely abort trigger.)

- **[MAJOR] No versioning/schema/provenance story across `.mcp`/`.mcs`/user-metadata.** The "large user-metadata carve-out" and `.mcs` "free-form JSON" are the integration channel to every consumer yet have no schema, required-fields contract, or compat rule, and no owner. `format_version=8` is declared "complete" / "an implementation micro-detail" while the plan simultaneously promises future non-breaking additions (u64/s64/f64, `.mcs` format, AVX-512, f16-native) — so the format *will* evolve with no negotiation/migration policy. `.mcs` references `.mcp` surfaces with no shared version/provenance link, so a segmentation can be silently sampled against a re-ingested (different) volume. **Fix:** define a version-negotiation/migration policy, a minimal `.mcs` schema with required provenance fields, and a `.mcp`↔`.mcs` provenance link checked at load.

- **[MINOR] xxh3 integrity is inconsistent between sparse and packed forms.** The sparse local file's per-chunk xxh3 is "over the stored bytes as written" — including in-slot page-alignment padding — but the packed remote is dense with no padding, so the bytes differ and the same hash value cannot apply. Either the "trivial sparse→packed serialization" must recompute a *different* hash (so it is not trivial, contradicting the everything-else memo's "no repack tool" claim) or the hash must be payload-only (contradicting the archive's "stored bytes as written"). The hash is also only verified by an offline tool, never on the hot path. **Fix:** scope xxh3 to chunk payload bytes (not stored layout), make it identical across sparse/packed forms, and resolve the repack-tool contradiction.

- **[MINOR] "Coordination is deterministic" silently depends on an unpinned worker count / partition geometry.** The entire test strategy leans on "values non-deterministic BUT coordination is a deterministic testable transform." But merge/dedup/apply order is a function of N workers, band partitioning, and IO fan-out width — none fixed — and the cache showed apply() order affects which generation wins at equal generations (M1). So coordination is deterministic only at *fixed* worker count/partition geometry. **Fix:** specify a canonical merge/apply order independent of worker count and partition geometry, or retract "coordination is deterministic" as a checkable property.

## 4. What the Plan Got RIGHT

This plan withstood adversarial pressure better than its finding count suggests — most "CRITICAL/MAJOR" attacks were downgraded or refuted because the plan had already handled the objection.

- **The BSP cache non-leak reframe is a genuine achievement, not a rename.** The original cache's real defect (a read-phase-mutating miss ring reaching into requester state) is actually eliminated: misses are per-requester local buffers, the cache mutates nothing during freeze. That is a real achieved property.
- **Render-now-refine-later is the correct interactive model**, and the plan adopted it deliberately. It neutralized the most-cited "frame stalls 100ms on cold fetch" attack — synchronous in-frame fetch (the implied alternative) would be strictly worse.
- **The 3-state occupancy map (DONT_KNOW/ALL_ZERO/REAL) is a strong design.** It makes "not yet fetched" safely distinct from "confirmed empty," is the authoritative occupancy oracle (correctly *not* SEEK_HOLE), enables zero-network air chunks, and forecloses the stale-zero-alias "black hole" entirely.
- **Static linear_slot_index + sparse mmap is the right load-bearing bet.** It survives the TB/EB sizing critique as the *same* problem at scale, and it genuinely bounds crash blast radius to in-flight chunks (no torn grown index).
- **The persistent write-through local mirror** correctly amortizes whole-chunk over-fetch to ~1× per session and makes the partial-fetch sacrifice defensible.
- **Air bit-exactness via an integer mask outside the float path genuinely survives fast-math** — the one codec invariant that is airtight, and correctly separated from the (unsound) residual-correction guarantee.
- **The platform seam (`mcpp::os`) correctly isolates** mmap/IO/threading primitives, making the "no synchronization in application code" claim defensible once reworded.
- **The codec variant-dispatch matrix does NOT explode** — measured at ~32 instantiations / sub-second compile / KB-scale binary. The plan correctly resisted pre-emptive type-erasure.
- **Mostly-honest about its own open items** — §8.8 already flags gating benchmarks, generator design, and golden-corpus as unresolved, which is why several "you ignored X" findings were downgraded to "you under-prioritized X."
- **The tracer's real access pattern (build-index-once, query read-only) already matches the static-tree model** — the plan correctly scoped incremental indexing out.
